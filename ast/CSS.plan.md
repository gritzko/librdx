#   ast/CSS — CSS-like selector for BASON trees

CSS-like query language for selecting nodes in BAST/BASON
trees. Used in URI fragments (`#fn.main`, `#fn:has(malloc)`).

##  Files

    ast/CSS.lex        ragel grammar (input to `lex`)
    ast/CSS.c.rl       generated by `lex CSS c` (+ manual _out label)
    ast/CSS.rl.c       generated by `ragel -C CSS.c.rl -o CSS.rl.c -L`
    ast/CSS.h          public API: types + CSSParse + CSSMatch
    ast/CSS.c          callbacks (CSSonXxx) + matcher

##  Step 1: lex grammar (`CSS.lex`)

Tokens the lexer must recognize:

    ws       = [ \t];
    ident    = [a-zA-Z_] [a-zA-Z0-9_\-]*;
    number   = [0-9]+;
    string   = ["] [^"]* ["] | ['] [^']* ['];
    glob     = ident [*];

    # pseudo-elements (kind names)
    Kind     = ident;               # fn, class, args, cmt, ...

    # name match (CSS class)
    Dot      = ".";
    Name     = ident | glob;        # .main, .BE*

    # combinators
    Child    = ">";
    Adjacent = "+";
    Sibling  = "~";
    # descendant = whitespace (implicit between selectors)

    # pseudo-classes
    Has      = ":has(";
    Not      = ":not(";
    CloseParen = ")";

    # attribute selectors
    OpenBracket  = "[";
    CloseBracket = "]";
    AttrOp   = "=" | "^=" | "$=" | "*=" | "~=";
    AttrName = ident;
    AttrVal  = string | ident;

    # line range (GitHub convention)
    Line     = "L" number ("-" number)?;

    Root     = (ws | Kind | Dot | Name | Child | Adjacent
               | Sibling | Has | Not | CloseParen
               | OpenBracket | CloseBracket | AttrOp
               | AttrName | AttrVal | Line)*;

Run: `lex CSS c` → `CSS.c.rl`, then add `_out:` label,
then `ragel -C CSS.c.rl -o CSS.rl.c -L`.

##  Step 2: query tree representation

The parsed selector is stored as a BASON tree in a buffer.
Query node types (vowels for containers, consonants for
leaves — reusing the BASON type scheme):

    # Containers (logic nodes)
    'A' = AND (implicit: compound selector)
    'O' = OR  (selector list, comma-separated — future)
    'E' = HAS (children must match inside target)

    # Leaves (match predicates)
    'S' = text match (substring in :has() for leaves)
    'F' = name match (.main → key="name", val="main")
    'T' = type/kind match (fn → val="E", class → val="I")
    'N' = line range (val = two u32: start, end)
    'R' = attribute match (key=op like "^=", val=pattern)
    'P' = combinator (val = ">", "+", "~", " ")

Example: `fn:has(> args:has(u8bp))` parses to:

    A                       (root AND)
    ├── T val="E"           (type = fn)
    └── E                   (HAS)
        └── A               (AND)
            ├── P val=">"   (child combinator)
            ├── T val="U"   (type = args)
            └── E           (HAS)
                └── S val="u8bp"  (text match)

The kind-name-to-BASON-type mapping table:

    fn      → 'E'
    class   → 'I'
    args    → 'U'
    obj     → 'O'
    block   → 'A'
    def     → 'F'
    cmt     → 'D'
    str     → 'G'
    num     → 'L'
    type    → 'T'
    kw      → 'R'
    pp      → 'H'
    punct   → 'P'

Stored as a static lookup in CSS.c; the lexer callback
`CSSonKind` translates on the fly.

##  Step 3: parser callbacks (`CSS.c`)

State struct:

    typedef struct {
        u8cs data;          // input selector string
        u8bp buf;           // output BASON query tree
        u64bp idx;          // BASON index stack
        u8 pending_comb;    // buffered combinator char
        int paren_depth;    // :has() / :not() nesting
    } CSSstate;

Callbacks (generated signatures from .lex):

    CSSonKind       → look up kind table, emit T leaf
    CSSonDot        → set flag: next Name is a .name match
    CSSonName       → emit F leaf (key="name", val=<ident>)
    CSSonChild      → set pending_comb='>'
    CSSonAdjacent   → set pending_comb='+'
    CSSonSibling    → set pending_comb='~'
    CSSonHas        → BASONFeedInto 'E', push paren depth
    CSSonNot        → BASONFeedInto 'E' with negation flag
    CSSonCloseParen → BASONFeedOuto, pop paren depth
    CSSonOpenBracket  → start attribute context
    CSSonCloseBracket → emit R leaf from buffered attr
    CSSonAttrOp     → buffer operator
    CSSonAttrVal    → buffer value
    CSSonLine       → emit N leaf with line range

Between tokens, whitespace means descendant combinator
(if between two kind/name tokens). The callback for ws
sets pending_comb=' ' unless another combinator is set.

##  Step 4: matcher (`CSSMatch`)

Public API:

    // Parse selector string into query BASON tree
    ok64 CSSParse(u8bp qbuf, u64bp qidx, u8cs selector);

    // Match query against BASON data tree
    // Output: bitset of matched byte ranges (line-granularity)
    // Same bitset pattern as BASTGrepNodes.
    ok64 CSSMatch(u8s out, u8cs bason_data, u8cs query,
                  int context_lines);

Matcher algorithm — single recursive pass of the data tree:

    struct css_frame {
        u8  data_type;       // BASON type of current data node
        u8cs data_name;      // name (F leaf) if seen
        // :has() pending flags:
        int  n_pending;
        struct {
            u8cs  pattern;   // what to find
            b8    matched;   // found it yet?
            b8    negated;   // :not(:has()) ?
            b8    direct;    // > inside :has() ?
        } pending[8];
        // sibling tracking:
        u8  prev_type;
        u8cs prev_name;
    };

    css_frame stack[64];     // ancestry stack
    int depth = 0;

Walk with BASONDrain/BASONInto/BASONOuto:

  1. **Enter container**: push frame, copy pending `:has()`
     predicates from the query tree for this level.

  2. **Visit leaf**: check text against all pending `:has()`
     patterns on ancestor frames. If match, set
     `ancestor.pending[i].matched = YES`.
     Also check direct type/name predicates.

  3. **Exit container**: resolve all pending `:has()` for
     this frame. If all satisfied (and `:not()` inverted),
     mark this node's byte range in the output bitset.
     Update parent's sibling state.

  4. **Sibling combinators**: on entering a node, compare
     with `stack[depth].prev_type/prev_name`. For `+`
     check previous sibling matches. For `~` check any
     prior sibling (set a sticky flag).

After the walk, expand the bitset by context_lines (same
as BASTGrepNodes) and output matching lines.

##  Step 5: integration

  - `be cat file.c#fn.main` — parse fragment as CSS
    selector, match against merged BASON, output matching
    regions with syntax highlighting.

  - `be grep #fn:has(malloc)` — CSS match on every file
    (trigram pre-filter on "malloc"), output matching lines.

  - `be diff file.c#fn.main` — restrict diff to matched
    regions only.

  - HTTP: `GET /file.c#fn.main` — fragment is client-side
    per RFC, but the query param `?sel=fn.main` could serve
    the same role server-side.

##  Step 6: tests

  - `ast/test/CSS.c` — table-driven: selector string +
    input BASON → expected matched line ranges.
  - Cases: kind match, name match, `:has()`, `:not()`,
    combinators (`>`, `+`, `~`, descendant), line ranges,
    attribute matchers, nested `:has()`.
  - Fuzz: `ast/fuzz/CSS.c` — random selector strings
    against fixed BASON trees, must not crash.
